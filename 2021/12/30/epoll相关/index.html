三种IO多路复用

select

  • 缺点
    （1）每次调用select，都需要把fd集合从用户态拷贝到内核态，这个开销在fd很多时会很大
    （2）同时每次调用select都需要在内核遍历传递进来的所有fd，这个开销在fd很多时也很大
    （3）select使用是bit 数组了，支持的文件描述符数量太小了，默认是1024
  • 优点
    (1) 跨平台。

poll

除了数据结构不一样以外，基本和select一样。

epoll

工作流程：

1. 用户进程调用epoll_create;
2. 内核创建readyList,红黑树;
3. 用户进程调用epoll_ctrl,传递监控的句柄(如Socket),以及在上面关注的事件;
4. 内核将句柄插入红黑树;
5. 内核的中断处理程序注册一个回调,如果红黑树中某个句柄的中断到了,
把它对应的事件放到ReadyList. 

--- 另一个流程 --- 
1. 用户进程调用epoll_wait
2. 内核从ReadyList返回可回调的事件.

epoll事件有两种模型，边沿触发：edge_triggered(et),level_trggered(lt);

问题

  • 1 为什么epoll要用红黑树，而不用哈希表之类的数据结构 
1 epoll要快速找到某一个句柄，候选实现：
    (1) 查找树 O(lgn)
    (2) 哈希表 O(1)
    (3) 跳表 近似O(lgn)
    而选红黑树不选用AVL树的原因：

内核源码中说(网上查到的自己没看)
    Red-black trees are similar to AVL trees, but provide faster real-time boundedworst case performance for insertion and deletion (at most two rotations andthree rotations, respectively, to balance the tree), with slightly slower(but still O(log n)) lookup time.  
    红黑树类似于AVL树，但提供更快的实时性，插入和删除的有界最坏情况性能（最多两个分别旋转和三次旋转以平衡树）。查找时间稍慢（但仍为O（log n））。    

1. 如果插入一个node引起了树的不平衡，AVL和RB-Tree都是最多只需要2次旋转操作，即两者都是O(1)   
    但是在删除node引起树的不平衡时，最坏情况下，AVL需要维护从被删node到root这条路径上所有node的平衡性，  
    因此需要旋转的量级O(logN)，而RB-Tree最多只需3次旋转，只需要O(1)的复杂度。    
2. 其次，AVL的结构相较RB-Tree来说更为平衡，在插入和删除node更容易引起Tree的unbalance，  
    因此在大量数据需要插入或者删除时，AVL需要rebalance的频率会更高。因此，RB-Tree在需要大量插入和  
    删除node的场景下，效率更高。自然，由于AVL高度平衡，因此AVL的search效率更高。    
3. map的实现只是折衷了两者在search、insert以及delete下的效率。总体来说，RB-tree的统计性能是高于AVL的。  

2 选树的原因
    (1) 频繁增删改查，哈希表需要预估空间大小，这个场景无法做到，
        间接影响响应时间，哈希表有扩容策略的，扩容数据需要移动，即使使用了一致性哈希算法，也难以满足非阻塞timeout时间限制。
    (2) 百万级别连接的时候，哈希表会变得很大，连接变少的时候就会浪费很大的内存。
    (3) 跳表，慢于红黑树，空间也高
  • 2 epoll 就一定比select快吗？
select我们知道需要将select数组注册到内核中，内核需要轮询所有描述符检查其是否就绪，开销很大。  
epoll模型虽然在用户空间也需要轮询所有返回的就绪描述符，这些描述符都是已经就绪的但并非其优势所在，原因在于只需要将感兴趣的文件描述符  
与事件注册到epoll里面的红黑树中,当事件触发的时候会调用其回调函数，回调函数负责把就绪的加入就绪链表，等用户调用epoll_wait。  

但是epoll一定会比select快吗?

  • 3 为什么epoll的et模式文件要设置成非阻塞
因为将任务分给线程执行的时候，最后一次读必会阻塞线程，写会阻塞吗？写也会阻塞，当发送的缓存区满了，send就会阻塞。

  • tips: 关于read和write阻塞非阻塞的区别

  • read :
    返回值：
    n > 0 正常读到
    n == 0 套接字关闭，读到fin
    n < 0

errno == EINTR 操作中断，可以继续读取  ；  
errno == EWOULDBLOCK 如果是非阻塞的，无数据会返回；  
errno == EAGAIN 表示过会再试 EAGAIN == EWOULDBLOCK；

所以ET模式下阻塞非阻塞的区别：
对于非阻塞来说，只要缓存区有数据，就会立即返回，而缓存区为空的时候非阻塞和阻塞会表现得不一样，阻塞模式会等待，非阻塞模式会立即返回-1，(errno == EAGAIN ||errno == EWOULDBLOCK)

  • write
    阻塞的write只有再缓存区放得下整个buff时才会返回，非阻塞write返回放得下的字节数并(errno == EAGAIN ||errno == EWOULDBLOCK),阻塞write正阻塞时，对面关闭了套接字也会立即返回，立刻将剩余缓存区填满，但下次调用会失败。

  • 4 epool模式下的oneshot

oneshot主要是为了解决一个套接字被两个线程处理的情况，每次事件触发后，就从红黑树删除这个监听socket,
由于ET需要循环读取所有数据，因此没太大问题，但是LT一般只读取一次，加上ONESHOT的话就再也读取不了后面的数据了，因此LT也需要循环读取直到EAGAIN
