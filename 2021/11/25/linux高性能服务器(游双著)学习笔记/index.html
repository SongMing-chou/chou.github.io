第一篇TCP/IP协议详解

DSN工作原理

什么是DNS

DNS就是域名系统，是因特网中的一项核心服务，用于实现域名和IP地址相互映射的一个分布式数据库，能够使用户更方便访问互联网，而不用去记住被机器直接读取的IP，
DNS服务器主要基于UDP运输协议，这样能尽最大能力交付不可靠数据连接，而不是使用TCP，这样的面向连接的可靠数据连接。

DNS解析流程

1.打开浏览器，输入一个域名www.chousm.com，操作系统先回检查自己本地host文件是否有这个网站映射关系，如果有，就先调用这个地址，完成域名解析。
2.如果本地host没有这个域名的映射，则查找本地DNS解析缓存，是否有这个网址的映射关系，若果有，直接返回，完成域名解析。
3.如果host与本地DNS解析器缓存都没有相应的网址映射关系，首先会找TCP/IP参数中设置首选DNS服务器，在此我们叫他本地DNS服务器，服务器收到查询后，如果查询的域名，包含在本地配置的资源中，则解析返回客户机，完成域名解析，具有权威性。
4.如果要查询的域名，不由DNS服务器区域解析，但该服务器已缓存此网址隐射关系。则调用这个IP地址映射，完成解析，但是不具权威性。
5、如果本地DNS服务器本地区域文件与缓存解析都失效，则根据本地DNS服务器的设置（是否设置转发器）进行查询，如果未用转发模式，本地DNS就把请求发至13台根DNS，根DNS服务器收到请求后会判断这个域名(.com)是谁来授权管理，并会返回一个负责该顶级域名服务器的一个IP。本地DNS服务器收到IP信息后，将会联系负责.com域的这台服务器。这台负责.com域的服务器收到请求后，如果自己无法解析，它就会找一个管理.com域的下一级DNS服务器地址(http://chousm.com)给本地DNS服务器。当本地DNS服务器收到这个地址后，就会找http://chousm.com域服务器，重复上面的动作，进行查询，直至找到www.chousm.com主机。
6、如果用的是转发模式，此DNS服务器就会把请求转发至上一级DNS服务器，由上一级服务器进行解析，上一级服务器如果不能解析，或找根DNS或把转请求转至上上级，以此循环。不管是本地DNS服务器用是是转发，还是根提示，最后都是把结果返回给本地DNS服务器，由此DNS服务器再返回给客户机。
解析顺序：

1）浏览器缓存；
2）系统缓存；
3）路由缓存；
4）ISP（互联网服务供应商）DNS缓存；
5）根域名服务器；
6）顶级域名服务器；
7）主域名服务器；
8）保存结果至缓存。(指本地服务器，以备下一次使用，同时反馈给客户端)。

第四章 tcp协议

TIME_WAIT状态

客户端连接在收到服务器的结束报文段之后，并没有就直接进入CLOSED状态，而是转移到TIME_WAIT状态，在这个状态，客户端连接要等待一段长为2MSL（最大报文生存时间)
TIME_WAIT状态存在的原因有两点：

  • 可靠地终止TCP连接；
  • 保证让迟来的TCP报文段有足够的时间被识别并丢弃，(不要让相似的连接接收上一个连接的数据，相似指相同的IP地址和端口号)。

    堵塞控制

    堵塞控制的最终受控变量是发送端向网络一次连续写入的数据量，我们称为SWND(send window)，实际的SWND值是RWND和CWND中的较小值者。

    HTTP请求

    1 GET： 申请资源，而不对服务器产生任何其他影响；
2 HEAD ：和GET方法类似，不过仅仅要求服务器返回头部消息，而不需要传输任何实际的内容；
3 PUT ：上传某个资源
4 DELETE:删除某个资源
5 TRACE 要求目标服务器返回原始HTTP请求的内容，它可以用来查看中间服务器(比如代理服务器)对HTTP的影响；
6 OPTION:查看服务器对某一个URL都支持那些请求方法。也可以把URL设置为*，从而获得服务器的所有请求方法；
7 CONNECT：用于某些代理服务器，他们能把请求的连接转化成一个安全通道；
8 PATCH：对某个资源部分修改。
    在这些方法中，HEAD,GET,OPTION和TRACE被视为安全的方法，只获得资源而不对服务器做任何修改；而POST,PUT,DELETE和PATCH则影响服务器上的资源。另一方面，GET,HEAD,OPTION,TRACE,PUT,DELETE等请求方法被认为是等幂的(idempotent)，即多次重复的请求和只发送一次该请求具有完全相同的效果，而post方法则不同，连续多次发送同样一个请求可能进一步影响服务器上的资源。

  • 短连接和长连接：Connection头部字段就是专门用于告诉对方一个请求完成之后如何处理连接的，短连接就是处理完客户的一个HTTP请求之后，WEB就关闭连接(主动，TCP)，此后同一客户如果再请求一个HTTP请求，就需要再次连接，而长连接与之相反。

  • 注意！！！，在所有头部字段之后，HTTP请求必须包含一个空行，以标识头部字段的结束，请求行和每个头部消息都必须以”\\“结束(回车符和换行符)：而空行则必须只包含一个“\\”，不能有其他字符，甚至是空白符。

HTTP状态码

1xx:            100 Continue： 服务器收到了客户端的请求行和头部消息，告诉客户端继续发送数据部分，
                           客户端通常首先要先发送Expect:100-continue头部消息字段告诉服务器自己还有数据要发。
2xx成功：        200 OK：   请求成功；
3xx重定向：      301 Moved Permanently:     资源被转移了，请求将被重定向；
                302 Found：     通知客户端资源能在其他地方找到，但需要GET方法来获取它；
                304 Not Modified：  表示被申请的资源没有更新，和之前获得的相同；
                307 Temporary Redirect : 通知客户端资源能在其他地方找到，与302不同的是，客户端可以使用原始  
                                        请求相同的方法来访问目标资源；
4xx客户端错误：   400 Bad Request: 通用客户请求错误；
                401 Unauthorized:请求需认证信息；
                403 Forbidden:访问被服务器禁止，通常是由于客户端没有权限访问资源；
                404 Not Found：资源没找到；
                407 Proxy Authentication Required:客户端需要先获得代理服务器的认证；
5xx服务器错误：   500 Internal Server Error:通用服务器错误；
                503 Service Unavailable:暂时无法访问服务器。

Cookie

由于HTTP协议是一种无状态的协议，即每一个http请求之间没有任何上下文关系。如果服务器后续HTTP请求时需要用到前面的HTTP请求相关信息，即服务器不知道用户上一次做了什么，客户端必须重传这些信息，这样就导致HTTP请求必须传输更多的数据。，这严重阻碍了交互式Web应用程序的实现。cookie就是用来解决比如如下场景：

  • 在典型的网上购物场景中，用户浏览了几个页面，买了一盒饼干和两瓶饮料。最后结帐时，由于HTTP的无状态性，不通过额外的手段，服务器并不知道用户到底买了什么，所以Cookie就是用来绕开HTTP的无状态性的“额外手段”之一。服务器可以设置或读取Cookie中包含的信息，借此维护用户跟服务器会话中的状态。在刚才的购物场景中，当用户选购了第一项商品，服务器在向用户发送网页的同时，还发送了一段Cookie，记录着那项商品的信息。当用户访问另一个页面，浏览器会把Cookie发送给服务器，于是服务器知道他之前选购了什么。用户继续选购饮料，服务器就在原来那段Cookie里追加新的商品信息。结帐时，服务器读取发送来的Cookie即可。Cookie另一个典型的应用是当登录一个网站时，网站往往会请求用户输入用户名和密码，并且用户可以勾选“下次自动登录”。如果勾选了，那么下次访问同一网站时，用户会发现没输入用户名和密码就已经登录了。这正是因为前一次登录时，服务器发送了包含登录凭据（用户名加密码的某种加密形式）的Cookie到用户的硬盘上。第二次登录时，如果该Cookie尚未到期，浏览器会发送该Cookie，服务器验证凭据，于是不必输入用户名和密码就让用户登录了。
  • Cookie的缺陷
  • Cookie会被附加在每个HTTP请求中，所以无形中增加了流量。
  • 由于HTTP请求中的Cookie是明文传递的，所以安全性成问题，除非使用超文本传输安全协定。
  • Cookie的大小限制在4 KB左右，对于复杂的存储需求来说是不够用的。

第二篇 深入解析高性能服务器编程

第五章linux网络编程基础API

主机字节序和网络字节序

字节序分大端字节序和小端字节序，大端字节序是指一个整数的高位字节存储在内存的低地址处，低字节存储在高地址除，小端字节序低位存储在低位，高位存储在高位。网络传输字节序使用的都是大端字节序。

  • 数据在传输之前都需要经过转换吗？
    “既然数据传输采用网络字节序，那在传输前应直接把数据转换成网络字节序，接收的数据转换成主机字节序再保存。”如果数据收发过程中没有自动转换机制，那当然需要程序员转换，这光想让人觉得可怕，难道真的要强求程序员做这些事情吗，实际上没必要，这个过程是自动的，除了向sockaddr_in结构体变量填充数据外，其他情况无需考虑字节序问题。

    两个专用socket地址

    struct sockaddr_in 
{
    sa_family_t sin_family; //地址族：AF_INET
    u_int_16_t sin_port;    //端口号
    struct in_addr sin_addr;//IPV4地址结构体，
}；
struct in_addr 
{
    u_int_32_t s_addr;  //ipv4地址，要用网路字节序来表示
}；
struct sockaddr_in
{
    sa_family_t sin6_family;//地址族，AF_INET6
    u_int_16_t sin6_port;//端口号
    u_int_32_t sin6_flowinfo;//流信息，应设置为0
    struct in6_addr sin6_addr;//ipv6地址结构体，见下面；
    u_int32_t sin6_scope_id;//scope id

    unsigned char sa_addr[16];//ipv6 地址，要用网络字节序来表示。
};

网络地址初始化

struct sockaddr_in addr;
char *serv_ip = "211.217.168.13";
char serv_port = "9190";
memset(&addr,0,sizeof(addr));
addr.sin_family = AF_INET;
addr>sin_addr.s_addr = inet_addr(serv_ip);//addr.sin_addr.s_addr = htonl(INADDR_ANY)自动获取ip地址。
addr.sin_port = htons(atoi(serv_port));

所有专用socket地址(以及sockaddr_storage)类型的变量在实际使用时都需要转化成通用socket地址类型sockaddr(强制转换)，因为所有SOCKET编程接口地址参数类型都是sockaddr.
网络编程步骤就不写了，哪本书都有。。。。
主要的过程：
//服务端
serv_sock = socket(PF_INET,SOCK_STREAM,0);//创建套接字
bind(serv_sock,(struct sockaddr*)&serv_addr,sizeof(serv_addr))//向套接字分配网络地址
listen(serv_sock,5)//队列大小5
clnt_sockfd = accept(serv_sock,(struct sockaddr*)&clnt_addr,&(sizeof(clnt_addr)));//接收一个连接
read()/write();
close(serv_sock);//关闭套接字描述符

//客户端
clnt_sock = socket(PF_INET,SOCK_STREAM,0);
connection(clnt_sock,(struct sockaddr*)&serv_addr,sizeof(serv_addr) );
read()/write();
close(clnt_sock);

数据读写

对于文件的read和write同样适用socket,但是socket编程接口提供了，几个专门socket数据读写的系统调用，他们增加了读写控制。其中用于TCP流读写系统调用的是：

#include<sys/types.h>
#include<sys/socket.h>
ssize_t recv(int sockfd,void*buf,size_t len,int flag);
ssize_t send(in socket,const void *buf,size_t len,int flag)

socket选项

如果说FCNTL系统调用是控制文件描述符属性的通用POSIX方法，那么下面两个系统调用则是专门用来读取和设置socket文件描述符属性的方法：

#include<sys.socket.h>
int getsockopt(int sockfd,int level,int option_name,void* option_value,socklen_t*restrict option_len);
int setsockopt(int sockfd,int level,int option_name,const void* option_value,socklen_t option_len);

值得提出的是，有些socket选项只能由accept调用listen系统调用前针对监听socket才有效，linux开发人员提供的解决方案是：对监听socket设置这些socket选项，那么accept返回的连接socket将会自动继承这些选项。而对客户端而言，这些socket选项则因该在调用connect函数之前设置，因为connect调用成功返回之后,tcp三次握手已经完成。
补充：

  • 为了接收数据，套接字内部有一个由字节数组构成的缓冲，如果这个缓冲被接收的数据填满会发生什么事，之后传递的数据是否会丢失？
    首先调用read函数从缓冲读取部分数据，因此缓冲并不总是满的，但是如果read函数读取速度比接收的数据慢，则缓冲可能被填满，此时套接字无法再接收数据，但即使是这样也不会发生数据丢失，因为传输套接字会根据接收端的状态传输数据，如果传输出错还会提供重传服务，因此面向连接的套接字除特殊情况不会发生数据丢失。
  • IO缓冲在每一个TCP套接字中单独存在。
  • IO缓冲在创建套接字的时候自动生成。
  • 即使关闭套接字也会继续传递输出缓冲中遗留的数据。
  • 关闭套接字将丢失输入缓冲中的数据内容。
